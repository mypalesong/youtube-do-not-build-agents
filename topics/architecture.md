# 수렴하는 에이전트 아키텍처

2024년 말부터 2025년 한 해 동안, 업계는 "에이전트 춘추전국시대"를 보냈다. 각 도메인마다 다른 에이전트, 각 프레임워크마다 다른 추상화. 그런데 Anthropic이 Claude Code와 Skills를 만들며 관찰한 것은 의외의 결론이었다.

> 에이전트의 아키텍처는 수렴하고 있다.

Zhang과 Murag은 이 수렴점을 네 개의 구성 요소로 정리한다.

## 네 개의 축

### 1. 에이전트 루프(Agent Loop)

에이전트의 핵심은 **컨텍스트를 관리하고, 토큰의 흐름을 제어하며, 다음 행동을 결정하는 순환 구조**다. 모델이 추론을 내놓으면, 그 결과를 읽고 도구를 호출할지, 답변을 낼지, 추가 정보를 요청할지를 판단한다.

이 루프는 프레임워크마다 표면은 달라도 구조는 같다. Claude Code, Cursor, Copilot, 그리고 수많은 에이전트 SDK가 결국 같은 흐름을 구현한다. **에이전트 루프는 오늘날의 "CPU 스케줄러"**에 가깝다.

### 2. 런타임 환경(Runtime Environment)

에이전트가 일을 수행하려면 **실행할 공간**이 필요하다. 파일 시스템, 코드 실행 환경(일반적으로 bash와 파이썬), 네트워크 접근. Claude Code에서 관찰된 결정적인 사실은 이렇다 -- **이 런타임만 있으면 거의 모든 작업이 가능하다.**

새로운 도구를 정의하는 대신, 에이전트에게 **셸과 파일 시스템을 내주면** 에이전트는 자기가 필요한 도구를 그 자리에서 만들어낸다. 이것이 "도구 호출" 패러다임을 떠나 "스크립트 실행" 패러다임으로 옮겨가는 근거다.

### 3. MCP 서버(Model Context Protocol)

런타임이 로컬 머신의 경계라면, MCP는 그 경계를 넘는 **외부 세계와의 연결 표준**이다. 데이터베이스, SaaS API, 사내 시스템. 이 모든 것을 에이전트가 일관된 방식으로 호출할 수 있게 해주는 프로토콜이다.

MCP는 **"무엇과 연결할 수 있는가"**를 표준화한다. 연결 자체의 규약이지, 어떻게 그것을 사용할지는 정의하지 않는다. 사용 방법은 다음 층 -- 스킬 -- 의 몫이다.

### 4. 스킬 라이브러리(Skills)

마지막 축은 **런타임에 동적으로 로드되는 도메인 전문성**이다. 조직의 암묵지, 업계의 절차, 특정 작업의 플레이북. 이것이 스킬이라는 형태로 저장되어 필요할 때만 꺼내진다.

이 네 축을 한 그림으로 보면:

```
┌─────────────────────────────────────┐
│          에이전트 루프              │  <- 의사결정 주도
└─────────────────────────────────────┘
            ↓ 실행
┌─────────────────────────────────────┐
│         런타임 환경                  │  <- 파일 시스템 + 코드 실행
│   ┌──────────────┬───────────────┐  │
│   │  MCP 서버들   │   스킬 폴더    │  │
│   │ (외부 연결)   │ (절차적 지식)  │  │
│   └──────────────┴───────────────┘  │
└─────────────────────────────────────┘
```

## MCP와 스킬의 역할 분담

가장 많이 혼동되는 지점이기도 하다. "MCP와 스킬은 뭐가 다른가?" 발표자들의 답은 간결하다.

- **MCP** = 외부 세계와의 **연결(connection)**
- **스킬** = 그 연결 위에서 수행되는 **절차(procedure)**

세금 처리를 예로 들자. 회계 시스템에 접속하고, 세무 당국 API를 호출하고, 은행 거래 내역을 가져오는 배관은 모두 **MCP 서버**의 역할이다. 이 연결들을 **어떤 순서로, 어떤 규칙에 따라 사용할지**를 기술한 것이 스킬이다.

두 층의 분리는 중요하다. **연결은 안정적이고 장기적이지만, 절차는 조직마다 다르고 시간마다 바뀐다.** 이 두 종류의 지식을 하나로 뭉치면 어느 한쪽이 경직되거나 유지 보수가 지옥이 된다. 분리해 두면 각자의 속도와 주인이 맞춰진다.

## 수직 시장을 여는 속도

이 아키텍처가 실무에서 가진 강력한 함의는, **새 도메인에 진입하는 속도**다. Anthropic은 이 조합으로 다음을 단기간에 출시했다.

- **금융 서비스 솔루션** -- 금융 데이터 API(MCP) + 재무 분석 플레이북(스킬).
- **생명과학 솔루션** -- 과학 문헌/실험 데이터베이스(MCP) + 연구 절차(스킬).

새 도메인마다 **새로운 에이전트**를 만들지 않는다. 범용 에이전트 루프와 런타임은 그대로 두고, 그 위에 **도메인 MCP + 도메인 스킬**만 얹는다. 같은 논리가 스타트업과 엔터프라이즈 모두에게 적용된다.

## 왜 이것이 "수렴"인가

Zhang과 Murag이 "수렴"이라는 단어를 쓰는 이유는, 이 네 축의 조합이 **누구든 처음부터 다시 설계할 이유가 없는 형태**가 되었다고 보기 때문이다.

- 에이전트 루프는 경쟁 프레임워크들이 닮아가고 있다.
- 런타임 환경은 결국 파일 시스템 + bash + 파이썬으로 수렴했다.
- MCP는 외부 연결의 사실상 표준이 되어가고 있다.
- 스킬은 도메인 지식을 담는 새 컨테이너로 자리를 잡았다.

이 수렴이 완성되면, 개별 개발자가 **새로운 에이전트를 만들 이유가 거의 사라진다.** 대신 이 구조 위에 **자기 도메인의 스킬(과 필요하면 MCP)만** 얹으면 된다. 그래서 발표 제목이 **"Don't Build Agents, Build Skills Instead"**인 것이다.

## 소프트웨어 엔지니어링처럼 다루기

마지막으로, 발표자들은 이 아키텍처가 자라나면서 **스킬 자체를 소프트웨어 엔지니어링 자산으로 다뤄야 한다**고 말한다.

- **테스트와 평가** -- 스킬이 의도한 대로 트리거되고 출력되는지 검증.
- **버전 관리** -- 스킬의 진화와 그로 인한 에이전트 행동 변화의 **계보(lineage)** 추적.
- **의존성 선언** -- 스킬이 어떤 다른 스킬, 어떤 MCP, 어떤 바이너리에 의존하는지 명시.

이 세 가지가 갖춰지면 스킬은 단순한 마크다운 파일을 넘어 **팀과 조직의 엔지니어링 자산**이 된다. 그리고 이 자산이 축적되는 속도가, 앞으로 조직이 에이전트를 얼마나 잘 활용하는지를 가른다.
