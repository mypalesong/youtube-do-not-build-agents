코드, 세계와의 인터페이스 ​

Anthropic이 Claude Code를 만들면서 얻은 가장 중요한 통찰은, 역설적으로 "코딩"에 대한 것이 아니었다. Zhang과 Murag은 그 통찰을 이렇게 요약한다.

코드는 단순한 사용 사례가 아니다. 코드는 **디지털 세계와 대화하는 보편 인터페이스(universal interface)**다.

이 관찰이 이 발표 전체의 기술적 토대를 이룬다. 스킬이 "도구(tool)"가 아닌 "스크립트"로 설계된 이유도, MCP가 스킬과 공존할 수 있는 이유도, 모두 이 명제에서 출발한다.

Claude Code는 사실 범용 에이전트다 ​

Claude Code는 "코딩 에이전트"라는 이름으로 출시되었지만, 실제 사용 패턴을 보면 그 이름이 좁은 것이었음이 드러난다.

  • 재무팀이 Claude Code에게 엑셀 데이터를 받아 피벗 리포트를 생성하게 한다.
  • 연구자가 CSV와 Python을 써서 데이터 분석을 돌린다.
  • 운영팀이 수백 개의 파일을 규칙에 따라 재배치하게 시킨다.

이 모든 것은 겉보기에는 서로 다른 도메인의 업무다. 그런데 공통점은 명확하다. 결국 코드가 매개한다. 파이썬 스크립트, 셸 명령, SQL, JSON 변환 -- 모두가 디지털 산출물을 만들거나 조작하는 보편 언어다.

이 관찰에서 Anthropic은 큰 의사결정을 한다. "도메인별 에이전트를 따로 만들지 말고, 코드를 다룰 줄 아는 범용 에이전트를 만들자." Claude Code는 코더만을 위한 도구가 아니라, 디지털 세계의 모든 작업을 코드라는 공통 언어로 해결하는 인터페이스다.

왜 코드가 보편적인가 ​

Zhang과 Murag은 코드가 갖는 몇 가지 속성을 짚는다. 이 속성들은 에이전트 설계에 결정적이다.

1. 결정적 실행(Deterministic Execution)
자연어는 해석의 여지가 있지만, 코드는 한 번 실행되면 같은 입력에 같은 출력을 낸다. 에이전트가 "다섯 건의 송장을 검증하라"는 지시를 매번 다르게 해석한다면 실무에 투입할 수 없다. 스크립트를 통해서만 일관성의 보장이 가능하다.

2. 구성 가능성(Composability)
스크립트는 다른 스크립트를 호출할 수 있고, 파이프라인을 이룰 수 있다. 에이전트가 수행하는 복잡한 작업은 사실 작은 스크립트들의 조합이다.

3. 관찰 가능성(Observability)
스크립트가 실행되면 로그와 출력이 남는다. 에이전트는 이 결과를 읽고 다음 행동을 결정할 수 있다. 자연어 응답과는 다른, 명확하고 기계 가독적인 피드백 루프가 생긴다.

4. 수정 가능성(Modifiability)
스크립트가 기대와 다르게 동작하면 에이전트는 파일을 열어 직접 고칠 수 있다. 이것이 "도구"와 "스크립트"를 가르는 결정적 차이다.

도구(Tool)가 아닌 스크립트(Script) ​

전통적인 에이전트 설계에서 "도구"는 에이전트가 호출하는 사전 정의된 함수였다. send_email(to, subject, body), calculate_vat(amount) 같은 것들. 이 방식에는 두 가지 구조적 한계가 있다.

  • 모호성 -- 도구의 설명문이 아무리 정교해도, 엣지 케이스에서 에이전트는 오판한다. "amount가 음수면 어떻게 하지? 통화가 JPY면?" -- 문서에 없으면 추측한다.
  • 블랙박스성 -- 도구의 내부를 에이전트가 보거나 고칠 수 없다. 한 번 잘못 정의된 도구는 다음 사용자가 또 같은 문제를 겪는다.

스킬 안의 스크립트는 이 두 문제를 동시에 해결한다. 코드가 자신의 정확한 동작을 설명한다. 그리고 잘못된 동작은 수정될 수 있다.

Zhang과 Murag은 이 점에서 흥미로운 주장을 한다. 에이전트의 능력을 늘리는 "올바른 방법"은 도구의 개수를 늘리는 것이 아니라, 스크립트가 놓일 파일 시스템과 실행 환경을 제공하는 것이다.

런타임이 이미 코드를 실행할 수 있다면 ​

이 통찰이 스킬 설계로 자연스럽게 이어진다. 만약 에이전트가 이미 코드를 실행하는 환경을 가졌다면(Claude Code가 그렇다), 스킬은 그저 그 환경에 던져둘 파일들이면 충분하다. 별도의 "도구 등록"도, "플러그인 인터페이스"도 필요 없다.

  • 스킬은 폴더다.
  • 에이전트는 폴더 안의 SKILL.md를 읽는다.
  • 필요한 스크립트를 실행한다.
  • 결과를 받아 다음 단계로 간다.

이 흐름에는 새로운 추상화가 없다. 운영체제의 파일 시스템과 프로세스 실행이라는, 이미 50년 넘게 검증된 메커니즘 위에 올라탄다. 이것이 스킬의 확장성과 단순성을 동시에 가능하게 한다.

MCP와의 관계 -- 대체가 아닌 보완 ​

발표의 후반부에서 이 명제는 Model Context Protocol(MCP)과의 관계로 확장된다.

  • MCP -- 외부 세계와의 연결. API, 데이터베이스, SaaS 도구.
  • 스킬 -- 그 연결 위에서 수행되는 절차적 전문성.

세무 처리를 예로 들면, MCP가 회계 시스템과 세무 당국 API에 연결하는 배관이라면, 스킬은 "이 회사의 분기 세무 처리는 이런 순서로 한다"는 플레이북이다. 둘은 대체재가 아니라 보완재다.

그리고 이 둘을 하나로 묶는 매개가 바로 코드다. MCP의 연결을 호출하는 것도, 스킬의 절차를 수행하는 것도, 결국 런타임 위에서 실행되는 코드 조각이다. 이것이 "코드는 디지털 세계의 보편 인터페이스"라는 명제의 완전한 형태다.
